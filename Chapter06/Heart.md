# Heart

## Purpose

ไฟล์นี้เก็บหลักการกลางสำหรับการสร้างข้อสอบจำลองปลายภาคให้มีคุณภาพสูง ใช้ได้กับหลายบท ไม่จำกัดเฉพาะเรื่อง revaluation

เป้าหมายไม่ใช่แค่ทำข้อสอบให้ "ยาว" แต่ต้องทำให้ข้อสอบวัดความเข้าใจเชิงบัญชีจริง วัดลำดับคิดจริง และทำให้คำตอบของนักศึกษาสะท้อนว่าเข้าใจบทนั้นเป็นระบบ

## Core Belief

ข้อสอบปลายภาคที่ดีต้องวัดพร้อมกันอย่างน้อย 4 ชั้น

1. `policy`
2. `measurement`
3. `recognition location`
4. `statement effect`

แปลเป็นภาษาง่าย:

- กิจการใช้นโยบายอะไร
- ตัวเลขวัดอย่างไร
- ผลกระทบเข้าที่ไหน เช่น `P/L`, `OCI`, `equity`, `asset`, `liability`
- สุดท้ายรายการนั้นไปปรากฏอย่างไรในงบการเงิน

ถ้าข้อสอบวัดได้เพียงการแทนสูตร ข้อนั้นยังไม่ดีพอสำหรับปลายภาค

## Universal Design Principles

### 1. Build around the chapter's real spine

ก่อนออกข้อสอบ ต้องหา "แกนแท้" ของบทให้ได้ก่อน

ตัวอย่างของแกนที่อาจพบในแต่ละบท:

- การรับรู้รายการ
- การวัดมูลค่าเริ่มแรก
- การวัดมูลค่าภายหลัง
- การด้อยค่า / การกลับรายการ
- การจำหน่าย / derecognition
- การแสดงรายการในงบ

ข้อสอบต้องวิ่งตามแกนของบท ไม่ใช่สุ่มหัวข้อย่อยมาวางติดกัน

### 2. One long chain is better than many disconnected asks

ข้อสอบปลายภาคควรเป็นเส้นเรื่องต่อเนื่อง

รูปแบบที่ดี:

`เริ่มต้น -> เหตุการณ์ระหว่างทาง -> เหตุการณ์เปลี่ยนสถานะ -> เหตุการณ์ปลายทาง -> งบการเงิน`

เหตุผล:

- ทำให้คำตอบต้นข้อมีผลต่อท้ายข้อ
- บังคับให้เด็ก carry forward ตัวเลขเอง
- แยกคนที่ "เข้าใจตรรกะ" ออกจากคนที่ "จำสูตรเป็นช่วง ๆ"

### 3. Difficulty should come from accounting logic, not random arithmetic

ความยากที่ดีต้องมาจาก:

- การเลือกหลักการที่ถูก
- การแยกผลกระทบของรายการ
- การรู้ว่ารายการไหนกระทบงบไหน
- การแยกสิ่งที่ "มีข้อมูล" ออกจากสิ่งที่ "ต้องบันทึก"

ไม่ควรทำให้ยากแค่เพราะ:

- ตัวเลขเยอะเกินจำเป็น
- เศษทศนิยมประหลาด
- คำสั่งวกวน

### 4. Use carry-forward dependency deliberately

ข้อสอบที่ดีต้องมีจุดที่คำตอบก่อนหน้าเป็นฐานของคำตอบถัดไป

เช่น:

- มูลค่าตามบัญชีปลายปีนี้เป็นต้นปีหน้า
- ค่าเสื่อมหลังปรับปรุงต้องใช้ฐานใหม่
- ยอดสะสมเดิมต้องมีผลต่อการรับรู้รอบถัดไป
- ยอดใน equity หรือ allowance ต้อง carry ไปใช้ต่อ

### 5. End with statement presentation whenever the chapter supports it

ถ้าบทนั้นเกี่ยวข้องกับผลกระทบในงบการเงิน ข้อสอบควรปิดด้วย:

- งบกำไรขาดทุนเบ็ดเสร็จบางส่วน
- งบแสดงฐานะการเงินบางส่วน
- บางครั้งรวมงบแสดงการเปลี่ยนแปลงส่วนของเจ้าของ

เหตุผล:

- ทำให้เด็กต้องรู้ว่ารายการจบที่ไหน
- กันการทำแบบ mechanical journal entry only

## How to Make a Question Hard in a Good Way

### 1. Mix routine years with event years

อย่าให้ทุกปีมี event ใหญ่หมด

ให้มี:

- ปีธรรมดา: คิดค่าเสื่อม / ตัดจำหน่าย / adjustment routine
- ปีพิเศษ: remeasurement / impairment / disposal / revision / error discovery

ปีธรรมดานี่เองที่หลอกนักศึกษาได้ดี เพราะมักลืมทำรายการที่ควรทำต่อเนื่อง

### 2. Include at least one “information is not automatically entry” trap

ตัวอย่างหลักการ trap แบบดี:

- มี fair value แต่ไม่ได้แปลว่าต้อง remeasure
- มี indicator แต่ยังไม่แปลว่าต้อง recognize ทันทีโดยไม่ test
- มีการเปลี่ยน estimate แต่ไม่ใช่ prior period error
- มีตัวเลขมูลค่าตลาด แต่ยังต้องดูนโยบายบัญชีก่อน

### 3. Use time as a source of difficulty

ความยากที่ดีมักมาจากเวลา:

- ซื้อไม่ใช่วันต้นปี
- พร้อมใช้งานไม่ตรงกับวันซื้อ
- ขายระหว่างปี
- ปรับปรุงระหว่างปี
- เปลี่ยน estimate กลางอายุ

เวลาทำให้เด็กต้องคิด "งวด" ไม่ใช่คิดแบบ snapshot

### 4. Put one concept interaction point in the middle

ข้อสอบที่ดีควรมีจุดที่ concept 2 เรื่องชนกัน

เช่นในหลายบทอาจเป็น:

- การวัดมูลค่าหลังรับรู้ + ค่าเสื่อมราคา
- การด้อยค่า + การกลับรายการ
- การเปลี่ยน estimate + การนำเสนอในงบ
- การปรับมูลค่า + การจำหน่าย

ตรงนี้คือจุดคัดคนเก่งกับคนท่องจำ

### 5. Use one independent side-case only when it earns its place

ถ้าจะใส่ side-case เช่น ที่ดินคู่กับอาคาร หรือสินทรัพย์อีกตัวหนึ่ง ต้องมีเหตุผลชัด

เหตุผลที่ถูก:

- เพื่อเปรียบเทียบตรรกะของสินทรัพย์ 2 ประเภท
- เพื่อแยก concept ที่มักถูกปนกัน
- เพื่อให้เห็น difference in recognition path

ไม่ควรใส่เพียงเพื่อทำให้โจทย์ดูยาว

## Instruction Writing Rules

### 1. Do not over-hint

คำสั่งที่แยกย่อยมากเกินไปจะกลายเป็นเฉลยวิธีทำ

หลีกเลี่ยงคำสั่งแบบ:

- ให้ตัดกับ surplus เท่าไร
- ให้รับรู้ใน P/L เท่าไร
- ให้ reverse prior loss ก่อน
- ให้พิจารณา ceiling อย่างไร

ถ้ายังอยากวัดจุดนั้น ให้ถามกว้างขึ้น:

- บันทึกรายการที่เกี่ยวข้องพร้อมแสดงการคำนวณประกอบ
- พิจารณาและบันทึกรายการที่เกิดขึ้น
- จัดทำงบการเงินบางส่วน

### 2. But do not become vague to the point of unfairness

ห้ามกำกวมเรื่องข้อเท็จจริงหลัก

โจทย์ต้องบอกให้ชัด:

- นโยบายที่กิจการใช้
- วิธีเสื่อมราคา / ตัดจำหน่าย
- มูลค่าคงเหลือ
- อายุการใช้งาน / อายุคงเหลือ
- วันพร้อมใช้งาน
- วันจำหน่าย
- วิธีที่ให้ใช้ หากต้องการทดสอบ mechanics โดยตรง

### 3. Separate mechanics assumptions from the main case

ถ้าจะทดสอบอีกวิธีหนึ่ง เช่น alternate method หรือ alternate treatment ให้ทำเป็นข้อสมมติแยก

หลัก:

- main case ต้องสม่ำเสมอ
- side assumption ไม่ควรทำให้เส้นเรื่องหลักเสีย

## What Makes a Strong Final Exam Set

ข้อสอบปลายภาคที่แข็งแรงควรมีคุณสมบัติเหล่านี้

- มี narrative เดียวชัดเจน
- มี carry-forward
- มีอย่างน้อย 1 จุดที่นักศึกษาที่จำสูตรจะพลาด
- มีอย่างน้อย 1 จุดที่ต้องแยก `measurement` ออกจาก `recognition`
- มีผลไปถึงงบการเงิน
- คำสั่งไม่เฉลยวิธีทำ
- ขอบเขตยังอยู่ในบท ไม่ล้นไปบทอื่นโดยไม่จำเป็น

## A Reusable Workflow for Future Chapters

เมื่อต้องสร้างข้อสอบจำลองบทใหม่ ให้ใช้ workflow นี้

1. อ่านโน้ตของบท แล้วหา `chapter spine`
2. อ่าน assignment/solution ของอาจารย์ แล้วหา `teacher style`
3. สรุป `recurring traps` ของบท
4. เลือกว่าจะใช้โครงแบบ
   - single long chain
   - long chain + side-case comparison
   - integrated multi-part with final presentation
5. วาง timeline ของเหตุการณ์
6. ใส่ 1-2 trap ที่มาจาก logic ของบทจริง
7. ปิดด้วย statement presentation
8. ตรวจว่าคำสั่งไม่ได้ใบ้เกินไป

## Validation Checklist Before Finalizing Any Mock Exam

- บทนี้กำลังทดสอบอะไรเป็นแกนหลัก
- ทุกเหตุการณ์อยู่ในขอบเขตบทหรือไม่
- ตัวเลขปลายปีหนึ่งส่งผลไปปีถัดไปหรือไม่
- มีปี routine ที่ไม่มี event ใหญ่หรือไม่
- มีจุดแยก `information` กับ `entry` หรือไม่
- มีจุดวัด `statement effect` หรือไม่
- คำสั่งเฉลยวิธีทำมากเกินไปหรือไม่
- ถ้านักศึกษาทำผิดช่วงต้น ผลจะ carry ไปผิดช่วงหลังหรือไม่
- ความยากมาจากตรรกะ ไม่ใช่จากความรกของโจทย์ใช่หรือไม่

## Final Standard

ข้อสอบจำลองที่ดีต้องทำให้นักศึกษาที่อ่านบทแบบผิว ๆ รู้สึกว่าตอบไม่ได้หมด แต่คนที่เข้าใจโครงของบทจริงจะสามารถไล่แก้ได้เป็นขั้น

เป้าหมายไม่ใช่ทำให้โจทย์ "โหดแบบไร้เหตุผล"

เป้าหมายคือทำให้โจทย์:

- ยากอย่างเป็นระบบ
- หลอกอย่างยุติธรรม
- วัดความเข้าใจจริง
- และสะท้อนสไตล์อาจารย์ให้ใกล้ที่สุด
