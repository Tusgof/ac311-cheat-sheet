AC311 Exam Sheet Web Style Guide

วัตถุประสงค์
- ใช้สำหรับแปลงไฟล์เนื้อหาเรียนที่จัดระเบียบแล้ว เช่น `.md` ให้เป็นเว็บ HTML หน้าเดียวที่อ่านสอบได้จริง
- เป้าหมายหลักคือ `เนื้อหาครบ`, `สแกนเร็ว`, `อ่านยาวได้ไม่ล้า`, และ `ใช้งานได้ดีทั้ง desktop และ mobile`
- ห้ามลดทอนสาระสำคัญของเนื้อหาเพื่อแลกกับความสวย

Core Style

1. บุคลิกของงาน
- บุคลิกหลักคือ `clean student notebook`
- อารมณ์รวมต้องสะอาด เนี้ยบ อบอุ่น เป็นระเบียบ และดูเหมือนชีทสรุปที่ตั้งใจทำมาอ่านก่อนสอบ
- ไม่ให้ความรู้สึกแบบ SaaS dashboard, landing page ขายของ, หรือ presentation site

2. โทนภาพ
- ใช้โทนครีม-น้ำตาลเป็นฐาน
- พื้นหลังหลักควรเป็น warm cream ไม่ขาวจัด
- หัวข้อหลักใช้สีน้ำตาลเข้ม
- เส้นแบ่ง ข้อความรอง และโน้ตรอง ใช้น้ำตาลกลางหรือน้ำตาลอมเทา
- accent color มีได้เพียง 1 สี และต้อง muted เท่านั้น เช่น olive อ่อนหรือทองหม่น
- หลีกเลี่ยง gradient แฟนซี, glassmorphism, และเงาหนัก

3. Typography
- ใช้ font stack แบบเดียวกับหน้า Investment Property:
  `Google Sans Text`, `Google Sans`, `Product Sans`, `Noto Sans Thai`, `sans-serif`
- ถ้าเครื่องไม่มี Google Sans ให้ fallback ไปที่ Noto Sans Thai โดยไม่ทำให้ภาษาไทยเพี้ยน
- หัวข้อหลักและ body ใช้ stack เดียวกันนี้ เพื่อให้บุคลิกทั้งหน้าตรงกับ Investment Property
- เนื้อความปกติต้องอ่านง่ายมากสำหรับข้อความยาวจริง
- keyword, หมายเหตุ, จุดหลอก, label ตาราง ต้องมีลำดับชั้นชัดผ่านน้ำหนักตัวอักษร ขนาด และ spacing
- code block และ ledger block ใช้ mono stack แยกจากเนื้อหาหลัก:
  `Google Sans Code`, `SFMono-Regular`, `Consolas`, `monospace`
- หลีกเลี่ยงฟอนต์ที่ดู tech, futuristic, หรือ corporate เกินไป

4. Layout
- หน้าแรกต้องเรียบและเข้าเรื่องเร็ว แต่ยังมี hero สั้น ๆ แบบ page opener ได้
- hero copy ต้องสั้นและบอกวิธีอ่านบททันที ห้ามใส่ข้อความอธิบายงานยาว ๆ
- ห้ามโชว์ metadata ที่ไม่ช่วยอ่านสอบใน hero เช่น Typeface, source note, หรือคำอธิบายว่าทำไฟล์นี้ขึ้นมาอย่างไร
- ถ้าต้องมี chip ใต้ hero ให้ใช้ 2-4 ชิ้นเท่านั้น และแต่ละชิ้นต้องสั้น เช่น มาตรฐาน, logic หลัก, Dr./Cr.
- ใช้ layout โปร่ง มีลมหายใจ และใช้ spacing เป็นเครื่องมือหลักในการแยกความสำคัญของข้อมูล
- ใช้ section แบบ notebook ที่จัดระเบียบดี แทน card grid
- โครงที่แนะนำคือ hero สั้น, sticky mini-nav, group, section, subsection, table/ledger/callout
- เส้นคั่น, grid เบา ๆ, paper texture, notebook cues ใช้ได้ แต่ต้อง subtle เท่านั้น
- ตารางควรมี wrapper สำหรับ horizontal scroll บนมือถือ และหัวตารางที่อ่านง่าย

4.1 Hero / Header Copy
- hero ของแต่ละบทต้องทำหน้าที่เป็น `exam orientation` ไม่ใช่คำโปรโมตหน้าเว็บ
- subtitle ใต้ชื่อบทควรบอกลำดับคิดของบทให้ใช้สอบได้ทันที เช่น `อ่านตามลำดับ: แยกประเภท -> รับรู้รายการ -> วัดมูลค่า -> โอน -> จำหน่ายและเปิดเผยข้อมูล`
- chips ใต้ hero ควรเป็น keyword ที่ช่วยจำแกนบท เช่น `TAS 40`, `Cost / Fair Value`, `Transfers + Disclosure`
- หลีกเลี่ยงข้อความที่ไม่ได้เพิ่มความเข้าใจ เช่น `Single-page exam sheet`, `ครบเนื้อหา + สแกนเร็ว`, `HTML ready`, `GitHub Pages`, หรือคำอธิบายว่าไฟล์นี้ใช้ทำอะไร
- ถ้า chip ใดไม่มีผลต่อการตัดสินใจทำข้อสอบ ให้ลบทิ้งหรือเปลี่ยนเป็นมาตรฐาน/โมเดล/ประเด็นข้อสอบที่ชัดกว่า
- หัวที่ดีควรตอบได้ใน 5 วินาทีว่า `บทนี้เรียนเรื่องอะไร`, `ควรอ่านตาม logic ไหน`, และ `keyword ใดต้องจำ`

5. Interaction
- interaction ต้องช่วยการอ่าน ไม่ใช่สร้างความหวือหวา
- ใช้ motion เบา เนี้ยบ และสม่ำเสมอ
- องค์ประกอบ interactive ที่เหมาะกับงานลักษณะนี้คือ:
  sticky navigation
  keyword highlight แบบนุ่ม ๆ
  hover/focus states สำหรับตารางและ callout
  collapsible บางส่วนในจุดที่เนื้อหายาวจริง
  target state ตอน jump ไป section
- ถ้ามี nav ให้มี active state ที่บอกตำแหน่งปัจจุบันได้ชัด
- ถ้ามีตารางยาว ให้ mobile scroll ได้โดยไม่พัง layout

Production Rules

1. ความถูกต้องของเนื้อหา
- ใช้ข้อมูลจากไฟล์ต้นฉบับหรือไฟล์ที่ผู้ใช้อัปโหลดเป็นฐานเท่านั้น
- ห้ามเติมข้อมูลวิชาการใหม่ถ้าไม่มีในต้นฉบับ
- ถ้าพบข้อมูลซ้ำ ให้รวมให้กระชับขึ้นได้ แต่ห้ามทำให้สาระหาย
- ถ้าพบข้อมูลกำกวม ให้คงสาระเดิมและจัดรูปแบบให้เห็นว่าเป็นหมายเหตุหรือข้อควรตรวจสอบ
- เป้าหมายคือ `จัดระเบียบ` ไม่ใช่ `สรุปย่อจนข้อมูลหาย`

2. การจัดโครงสร้างเนื้อหา
- ควรจัดเนื้อหาเป็นกลุ่มใหญ่ 5-7 กลุ่มตาม flow การอ่านสอบจริง
- หลีกเลี่ยงการปล่อย section ย่อยจำนวนมากเรียงยาวโดยไม่มีการจัดกลุ่ม
- ภายในแต่ละกลุ่ม ควรเรียงจาก:
  ภาพรวม / หลักคิด
  เส้นแบ่งหรือเกณฑ์ตัดสิน
  ตัวอย่าง
  หลุมพราง / ข้อสอบชอบหลอก
- ถ้าไฟล์มีหลายหัวข้อ ไม่จำเป็นต้องคงลำดับเดิมแบบดิบ หากจัดใหม่แล้วเข้าใจง่ายกว่า

3. โครงสร้างของแต่ละพาร์ท
- ในแต่ละพาร์ทให้ใช้ลำดับต่อไปนี้เมื่อเหมาะสม:
  ใจความหลัก
  keyword สำคัญ
  อธิบายการบันทึกบัญชี
  ตารางเปรียบเทียบ
  สรุปสั้น
  ข้อเพิ่มเติม
  จุดหลอก / จุดสับสน / สิ่งที่มักออกสอบผิด
- ถ้ามีข้อมูลเปรียบเทียบกันชัดเจน ให้แปลงเป็นตาราง
- ถ้ามีขั้นตอนบันทึกบัญชี ให้จัดให้อ่าน flow ง่ายและเห็น debit/credit ชัด
- ถ้ามีข้อยกเว้น เงื่อนไข หรือ trap ให้ทำเป็น callout ที่เด่นพอดี

4. Navigation
- ควรมี navigation 2 ชั้น:
  top navigation สำหรับกลุ่มหลัก
  local navigation ภายในแต่ละกลุ่มสำหรับ section ย่อย
- ทั้ง top nav และ local nav ควรมี active state ที่บอกตำแหน่งการอ่านปัจจุบัน
- ถ้ามี sticky navigation ต้องตั้ง scroll offset หรือ `scroll-margin-top` ให้หัวข้อไม่ถูกบัง

5. Visual Hierarchy
- ต้องมี hierarchy ชัดระหว่าง:
  group
  section
  subsection
  prose body
- แต่ละกลุ่มควรมี group header ที่ระบุชื่อกลุ่มและคำอธิบายสั้น
- สามารถใช้ theme เบา ๆ ต่างกันตามชนิดเนื้อหา เช่น concept / accounting / disclosure / review ได้ ถ้าช่วยให้สแกนง่ายขึ้น
- กล่องย่อยควรแยกประเภทอย่างน้อยเป็น:
  summary
  note
  example
  trap
  accounting entry

6. Spacing
- spacing ต้องถูกออกแบบเป็นระบบ ไม่ใช่ปรับรายจุด
- อย่างน้อยควรแยกระดับ spacing เป็น:
  ระหว่างกลุ่ม
  ระหว่าง section
  ระหว่าง subsection
  ระหว่าง paragraph / list / table / code block
- บนมือถือให้ลด padding และ gap ลงอีกระดับ แต่ยังต้องคงลมหายใจของหน้าไว้

7. ตาราง
- ตารางต้องอ่านง่ายมากและรับภาระเนื้อหาจริงได้
- ความหนาแน่นของตารางควรกระชับกว่าตารางเว็บทั่วไปเล็กน้อย
- ใช้ cell padding, line-height, และขนาดตัวอักษรที่เหมาะกับข้อมูลยาว
- ถ้าตารางยาว ควรใช้ sticky table header
- ควรมี zebra rows แบบบางมาก หรือ state ที่ช่วยไล่อ่านแถวได้ง่ายขึ้น
- ถ้ามีข้อมูลเชิงตัวเลข ควรใช้แนวคิดแบบ `tabular-nums` เพื่อให้เทียบตัวเลขง่าย

8. ตารางบนมือถือ
- ตารางต้องรองรับ horizontal scroll ได้อย่างเรียบร้อย
- ควรมี visual hint ว่ายังเลื่อนได้ เช่น edge fade หรือ shadow เบา ๆ ที่ขอบ
- หลีกเลี่ยงตารางที่บีบจนตัวหนังสือเละหรือคอลัมน์พังบนจอเล็ก

9. Accounting Blocks
- บล็อก `Dr./Cr.` หรือ ledger ต้องมีภาษาภาพต่างจาก prose ปกติชัดเจน
- ถ้าบล็อกยาวมาก สามารถใช้ collapsible ได้
- แต่ค่าเริ่มต้นของหน้าโดยรวมต้องยังอ่านต่อเนื่องได้ดี
- บล็อกลักษณะนี้ต้องยังอ่านบนมือถือได้โดยไม่แตก layout

10. Mobile Reading Flow
- ต้องออกแบบ mobile ก่อนอ่านยาวจริง ไม่ใช่แค่ย่อ desktop ลงมา
- local navigation ของแต่ละกลุ่มบนมือถือควรเป็น horizontal chips มากกว่าการ wrap หลายบรรทัด
- ระยะห่าง การเรียงตัวของ prose และขนาดตาราง ต้องช่วยให้เลื่อนอ่านต่อเนื่องได้

11. Motion และ Feedback
- motion ต้องช่วย hierarchy และ orientation เท่านั้น
- keyword highlight, nav active transition, และ hover/focus states ควร subtle
- ห้ามใช้ motion ที่ทำให้หน้าดูเหมือน showcase หรือ presentation มากกว่า notebook
- เมื่อกด jump ไป section ควรมี target state หรือ focus cue บาง ๆ ให้รู้ว่ามาถึงจุดไหนแล้ว

12. Output และ Workflow
- ผลลัพธ์ต้องเป็นไฟล์ HTML ที่เปิดได้ทันที
- CSS และ JavaScript ต้องอยู่ในไฟล์เดียวหรือจัดอย่างเรียบร้อย
- ถ้ามี generator สำหรับสร้าง HTML จาก Markdown ให้แก้ที่ generator เป็นหลัก ไม่ควรแก้เฉพาะ HTML ปลายทาง
- งานสุดท้ายต้องพร้อมใช้งานจริงด้วยเนื้อหาที่ผู้ใช้อัปโหลด

AC311 HTML Final Checklist

ใช้ checklist นี้ก่อนปิดงานทุกบท

1. Source และ Content Fidelity
- อ่านไฟล์ต้นฉบับครบแล้ว ไม่ใช่ดึงเฉพาะหัวข้อใหญ่
- เนื้อหาสำคัญจากต้นฉบับยังอยู่ครบ
- ไม่มีการเติมข้อมูลวิชาการใหม่ที่ไม่ได้อยู่ในต้นฉบับ
- จุดที่รวมข้อมูลซ้ำแล้ว ยังไม่ทำให้ logic หรือเงื่อนไขหาย
- ถ้ามีข้อความกำกวมหรือควรตรวจสอบ ต้องจัดเป็น note ไม่ใช่แต่งเพิ่มเอง

2. Markdown-to-HTML Pipeline
- HTML สร้างจาก Markdown หรือ source ที่กำหนดจริง ไม่ใช่ snapshot เก่าที่ค้างอยู่
- ถ้ามี generator ต้องแก้ generator ก่อน แล้วค่อย regenerate HTML
- หลัง regenerate แล้ว HTML ยังเปิดได้ทันที
- ไม่มีข้อความเพี้ยน, เครื่องหมายคำถาม `???`, หรือ mojibake จาก encoding
- title, meta, heading, nav และเนื้อหาแสดงภาษาไทยถูกต้อง

3. Information Architecture
- เนื้อหาถูกจัดเป็นกลุ่มใหญ่ตาม flow การอ่านสอบจริง
- แต่ละกลุ่มมีหน้าที่ชัด ไม่รวมหลายเรื่องปนกันจนสแกนยาก
- ลำดับภายในกลุ่มไหลจากหลักคิด ไปเกณฑ์ตัดสิน ไปตัวอย่าง ไปจุดหลอก
- บทท้ายหรือ review section ไม่ซ้ำกับ checklist แบบไม่จำเป็น
- ถ้ามี lifecycle หรือ roadmap ควรวางท้ายเพื่อใช้ทวนภาพรวม

4. Section Structure
- ทุก section มีหัวข้อที่อ่านแล้วรู้ทันทีว่าพูดเรื่องอะไร
- เนื้อหาใน section ไม่เป็นย่อหน้ายาวเกินไป
- จุดสำคัญถูกแตกเป็น bullet, table, callout หรือ ledger block ตามความเหมาะสม
- ตัวอย่างและข้อยกเว้นอยู่ใกล้หลักการที่เกี่ยวข้อง
- trap / จุดสับสน / ข้อสอบชอบหลอก ถูกแยกให้เห็นชัด

5. Navigation
- top navigation พาไปกลุ่มหลักได้ครบ
- local navigation พาไปหัวข้อย่อยภายในกลุ่มได้ครบ
- active state ของ nav ใช้งานได้และไม่รบกวนสายตา
- กด jump แล้วหัวข้อไม่ถูก sticky nav บัง
- บนมือถือ nav ไม่กินพื้นที่แนวตั้งมากเกินไป

6. Visual Hierarchy
- hierarchy ระหว่าง group, section, subsection, prose body ชัดเจน
- keyword สำคัญสแกนเจอได้ทันที
- label ของ callout และตารางไม่แข่งกับหัวข้อหลัก
- accounting entries หรือ Dr./Cr. block ดูต่างจากเนื้อความปกติ
- สีและน้ำหนักตัวอักษรช่วยอ่าน ไม่ใช่แค่ตกแต่ง

7. Tables
- ตารางทุกชุดอ่านง่ายบน desktop
- ตารางบนมือถือ horizontal scroll ได้โดย layout ไม่พัง
- มี visual hint ว่ายังเลื่อนได้ หากตารางกว้างกว่าหน้าจอ
- padding และ line-height ของตารางไม่หลวมเกินไปและไม่แน่นเกินไป
- ถ้ามีตารางยาว ควรมี sticky header หรือวิธีช่วยไม่ให้หลงคอลัมน์

8. Mobile Reading Flow
- เปิดบนจอเล็กแล้วยังอ่านต่อเนื่องได้
- spacing บนมือถือถูกลดอย่างพอดี ไม่แน่นจนเหนื่อย
- local nav บนมือถือไม่ wrap หลายบรรทัดจนดันเนื้อหาลงมากเกินไป
- ledger/code/table ไม่ล้นจนทำให้หน้าเสียรูป
- tap targets ของ navigation และ collapsible กดง่ายพอ

9. Motion และ Interaction
- motion ทั้งหมดช่วย orientation หรือ readability
- keyword highlight ไม่แรงจนรบกวนการอ่าน
- hover/focus state ของตารางและกล่องสรุปช่วยให้ไล่อ่านง่ายขึ้น
- collapsible ถ้ามี ต้องสื่อ affordance ชัดว่าเปิด/ปิดได้
- เคารพ reduced motion ถ้ามี animation หรือ transition หลายจุด

10. Visual Finish
- โทนรวมยังเป็น clean student notebook
- พื้นหลัง เส้น grid และ paper texture ต้อง subtle
- ไม่มีเงาหนัก, gradient แฟนซี, glassmorphism, หรือ dashboard feel
- spacing ระหว่างกลุ่ม/section/subsection/prose เป็นระบบ
- หน้าสุดท้ายไม่ดูเหมือน append ทิ้งไว้เฉย ๆ ต้องปิดงานเป็นระเบียบ

11. Final Verification
- จำนวน group และ section ตรงกับที่ตั้งใจ
- nav link ทุกตัวชี้ไป section ที่มีจริง
- ไม่มี quick review หรือ section เก่าที่ผู้ใช้สั่งลบทิ้งหลงเหลืออยู่
- ไม่มีเนื้อหาเก่าที่ถูกแทนที่แล้วหลงเหลือใน HTML
- เปิด HTML หลัง regenerate แล้วตรวจด้วยตาอย่างน้อย 1 รอบ
